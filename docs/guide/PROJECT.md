## 技術基盤の恒久前提

### DB / 接続

- DB は Neon (PostgreSQL 16) を前提にする。
- アプリ本体は pooled 接続、migration は direct 接続を使い分ける。

環境変数:
- `DATABASE_URL`: pooled 接続
- `DATABASE_URL_UNPOOLED`: direct 接続

### 検索

- P0 検索は `ILIKE` ベースとする。
- Neon 上で使える検索拡張は後段で再評価する。
- `pg_bigm` 前提で設計しない。

### DB 未接続時の扱い

- `DATABASE_URL` 未設定でもアプリ全体は起動可能であることを維持する。
- DB 必須 API はクラッシュではなく `503 database unavailable` を返す。
- UI 実装と DB 実装を疎結合に保つ。

## ソース取得の恒久方針

### 基本方針

- 収集元は `Google Alerts` を母集団とし、重要ソースを限定追加する。
- 重要ソースを自動取得する場合は、取得手段を制限する。

自動取得対象にしてよい条件:
- 公式 RSS / Atom がある
- 公式 API がある
- 利用条件が明示された公開 feed である
- ログイン不要の公開面である
- 保存・再配布の扱いが現行設計と矛盾しない

自動取得対象にしないもの:
- 規約が曖昧なスクレイピング依存ソース
- robots や利用条件の確認が面倒なソース
- 表示・再配布条件が読みにくいソース

初期方針:
- 初期投入する重要ソースは、公式 RSS / Atom / API がある範囲に限定してよい。

## Home 実装の恒久方針

### 実装順

- Home 画面は、モック UI を先に作り、後からデータを外だしする方式を許容する。
- ただしモックを残したまま見落とさないよう、可変項目は台帳化する。

### 非モック化

- 可変項目の棚卸しは `docs/imp/non-mock-ledger.md` を正とする。
- `取得元` と `非モック化ステータス` を持たない可変項目は増やさない。
- 新しい UI セクションを作るときは、必要に応じて台帳へ追記する。

### 表示優先順位

- P0 は表示速度を優先する。
- 検索 UI は Enter / ボタン submit を基本とする。
- 行動ログは UX を阻害しないことを優先し、必要なら表示後非同期送信を許容する。

## Topic Group の扱い

- Topic Group はプロダクト上の重要概念だが、最終遷移方式は未確定である可能性がある。
- そのため、P0 実装では「後で差し替えやすい暫定導線」を許容する。
- Topic Group の概念自体は固定し、最終 UI は `implementation-wait.md` で管理する。

固定前提:
- 同ジャンル内コサイン類似度ベース
- クロスジャンル統合なし

---

## 管理画面の考え方

- 管理画面は公開面より先に肥大化させない。
- P0 では運用開始に必要な最小機能だけを対象にする。
- フィード管理、要約監査、ジョブ確認は候補として妥当だが、境界の確定は `implementation-wait.md` で行う。

---

## 実装時の確認事項

- 仕様の恒久判断か、一時的な進捗メモかを先に切り分ける。
- 恒久判断なら `docs/guide/` または `docs/spec/` に残す。
- 未確定なら `docs/imp/implementation-wait.md` に残す。
- 一時的な引き継ぎなら `docs/imp/imp-hangover.md` に残す。
