# COMMON.md - 全エージェント共通ルール

最終更新: 2026-03-20

> このファイルは claude / codex / gemini の各エージェントが共有する共通ルールの SSOT。
> エージェント固有の設定は各 guide ファイルに置き、このファイルを重複させない。

---

## 1. 文字コードと読解ルール

1. 日本語を含むファイルは毎回 UTF-8 前提で読むこと。
2. 読み取り結果が不自然な場合は、ファイル破損ではなく文字コード起因を先に疑う。
3. 文字化けした表示のまま解釈して進めない。UTF-8 で再読込してから作業する。
4. UTF-8 で再確認せずに guide や spec を書き換えない。

---

## 2. docs 更新方針

1. `docs/` 配下の追記・更新は、原則として日本語で読める形で残す。
2. 一時メモ、進捗整理、判断待ち、運用メモも日本語で読み返せることを優先する。
3. 英語の識別子、API 名、外部サービス名、テーブル名、カラム名はそのままでよい。
4. 説明本文まで英語に逃がさず、日本語で意味が追えるように書く。
5. 方針変更・運用変更は `implementation-plan.md`、`imp-status.md`、必要に応じて `implementation-wait.md` や `docs/spec` に反映する。

---

## 3. ソースコードのコメント方針

1. コメントは最小限でよい。
2. コメントが必要な箇所では、日本語で簡潔に意図を書く。
3. 自明な処理説明（代入・ループ・標準的なパターン）のコメントは不要。
4. 複雑な分岐、運用制約、後続で誤読しやすい箇所だけにコメントを残す。

---

## 4. 検証優先ルール

1. 実装後は型・主要フロー・影響範囲の順で確認する。
2. 失敗時はすぐ修正に飛ばず、原因と再現条件を切り分ける。
3. 検証していない項目は、確認済みのように書かない。

---

## 5. タスク完了時の更新ルール

1. コードや挙動を変更したら、タスクを離れる前に `docs/imp/implementation-plan.md` の現況を更新する。
2. 実際に終わったこと、確認できたこと、残件は `docs/imp/imp-status.md` に追記する。
3. 判断が未了のまま終える場合は、必ず `docs/imp/implementation-wait.md` に論点と必要な判断を書く。
4. 今回やり切らない確認、障害試験、運用宿題は `docs/imp/imp-hangover.md` に残す。
5. 恒久仕様が変わった場合は `docs/spec/` か `docs/guide/PROJECT.md` を更新し、進捗メモだけで済ませない。
6. docs 更新なしでタスクを閉じてよいのは、コード・挙動・判断が何も変わっていない場合に限る。

---

## 6. タスク中断時の更新ルール

1. 中断時点での到達点、未実施、次の一手を `imp-status.md` または `imp-hangover.md` に残す。
2. 途中変更がある場合は、どこまで反映済みかを明確に書く。
3. 失敗した試行や却下した案も、再試行コストが高いものは簡潔に残す。

---

## 7. サブエージェント・ツール委託時のルール

1. サブエージェントや別ツールに実装を委託する場合、依頼プロンプトに **必ず** 以下を含めること：
   ```
   実装完了後、同一ターンで以下を更新すること：
   - docs/imp/ 配下の該当ファイル（実装反映・残件・判断待ち）
   - 仕様変更があれば docs/spec/ 配下も
   ```
2. サブエージェントはプロジェクト固有のルールを持たないため、明示しない限り docs は更新されない。
3. 更新指示を含め忘れた場合は、その後のターンで自分で補完する。

---

## 8. 技術基盤の共通前提

- **DB**: Neon (serverless PostgreSQL 16) — アプリ本体は pooled 接続、migration は direct 接続
- **認証**: Firebase Auth (Admin SDK + Client SDK) — `auth.uid()` ベースの Supabase 型 RLS は採用しない
- **AI**: Gemini Flash API (primary) / OpenAI (fallback)
- **フレームワーク**: Next.js 15 + TypeScript (strict)
- **スタイル**: Tailwind v4

### Next.js 15 Server Actions の注意点

- 回復可能な入力エラーを `throw` しない。`return { error: '...' }` 形式で返す。
- `DATABASE_URL` 未設定でもアプリ起動可能を維持する。DB 必須 API は `503` を返す。

## 9. このプロジェクトの docs 役割分担

| パス | 役割 |
|---|---|
| `docs/guide/` | 全エージェント共通ルール + プロジェクト固有の恒久ルール |
| `docs/spec/` | 仕様・アーキテクチャの SSOT（実装前提） |
| `docs/imp/implementation-plan.md` | 現在の実装計画と実装順 |
| `docs/imp/implementation-wait.md` | 未確定事項の集約 |
| `docs/imp/imp-hangover.md` | セッション引き継ぎ専用 |
| `docs/imp/imp-status.md` | 実施済み作業の記録 |
| `docs/imp/non-mock-ledger.md` | UI 非モック化対象台帳 |

原則: **恒久ルールは guide / 仕様は spec / 進捗は imp**

---

## 10. サマリ依頼時の扱い

1. ユーザーが「サマリ」を求めたときは、自由文より表を優先する。
2. 既定フォーマットは `docs/guide/.agent/workflows/refactor-summary.md` の workflow に従う。
3. 文章要約は短くし、行数の増減と区分別合計・総合計を優先して示す。
4. 表は Markdown table ではなく、`text` コードブロック内の等幅表を優先する。
