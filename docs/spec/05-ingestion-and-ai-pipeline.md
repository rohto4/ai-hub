# 収集・要約・批評パイプライン仕様（v2）

最終更新: 2026-03-06

## 1. フロー全体

1. RSS 取得
2. URL 正規化
3. 生データ保存
4. 重複判定（同ジャンル hash）
5. 本文抽出
6. AI 解析（要約/批評/タグ）
7. Topic Group 判定
8. 保存 + ランキング再計算キュー投入

## 1.1 収集元戦略（ユーザーに検索語を要求しない）

初期導入:
1. Google Alerts RSS（広域キーワード）
2. 公式技術ブログ RSS（OpenAI, Google AI, Anthropic, Meta AI, Microsoft AI など）
3. 主要 AI ニュース/まとめ系 RSS（英語・日本語の品質管理済みリスト）
4. YouTube 公式チャンネル更新（動画ジャンル用）

運用ルール:
1. 検索語は運営側が管理し、ユーザー入力不要
2. `seed keyword set` を 2 層で管理
   - Core: LLM/Agent/Inference/RAG/Safety 等
   - Trend: 週次で追加削除（新語対応）
3. 収集漏れ監視:
   - 週次で「後追い発見率」を計測しキーワード更新

## 2. URL 正規化ルール

1. `utm_*`, `fbclid`, `gclid` 除去
2. 末尾 `/` の統一
3. `http` -> `https` 優先
4. 短縮 URL は最終到達先へ展開

## 3. 同ジャンル重複検出

1. 正規化本文で SHA-256 生成
2. `(primary_genre, content_hash)` 一致で重複扱い
3. 重複時は `source_items.status = duplicate`

## 3.1 同ジャンル内の Topic Group 判定

方針:
- ジャンル横断（動画/ブログ/公式の束ね）は行わない
- 同ジャンル内で完全一致に近いトピックのみ統合する

判定ロジック:
1. 同ジャンル内で要約埋め込みのコサイン類似度を計算
2. 類似度 0.8 以上のみ自動統合
3. 0.8 未満は別記事として個別表示（統合しない）
4. レビューキューは設けない（統合できなければ別表示で許容）

備考:
- ベータ版では統合漏れより誤統合を避けることを優先する
- 閾値は運用データを見て調整可能

## 4. 100文字要約プロンプト設計

必須要素:
1. 何ができるか
2. なぜ凄いか（差分）
3. 誰向けか

禁止:
1. 100文字超過
2. 根拠なき誇張
3. 用語過密

補足:
1. 保存は `summary_100` を必須
2. 200/300 は拡張表示向けに同時生成可能

## 5. 批評生成設計

出力:
1. `innovation_delta`
2. `hype_risk`
3. `target_audience`
4. `evidence_strength`

判定基準:
1. 既存比較の明示性
2. 具体的な性能根拠
3. 限界の記載有無

## 6. コスト最適化

1. 重複判定後に AI 実行
2. 同 `content_hash` は結果再利用
3. 軽量モデルで一次要約、必要時のみ高性能モデル
4. 失敗リトライは指数バックオフ
5. ドメイン信頼度の高い定番ソースは軽量パス優先
6. 通知候補記事のみ高品質パスで再要約

## 7. 品質管理

1. 文字数検証（保存前）
2. 禁止語辞書チェック
3. 日次サンプル監査
4. 低品質検出時の再生成

## 8. 失敗時ハンドリング

1. 本文抽出失敗:
   - タイトル要約 + 要再処理フラグ
2. AI タイムアウト:
   - キュー再投入（最大 3 回）
3. 永続化失敗:
   - DLQ（Dead Letter Queue）へ退避

## 9. 初期キーワードバンドル（提案）

1. 基盤技術:
   - `large language model`, `LLM`, `inference`, `model optimization`, `RAG`
2. プロダクト/運用:
   - `AI agent`, `copilot`, `workflow automation`, `prompt engineering`
3. ガバナンス:
   - `AI safety`, `alignment`, `AI regulation`
4. 生成系実務:
   - `image generation`, `video generation`, `voice AI`

注記:
- 日本語キーワードは英語同義語セットとペアで運用する。
